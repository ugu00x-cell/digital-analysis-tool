## コード品質基準（Python）
以下のルールを、Pythonコードを書くときは常に適用する：
- 関数には必ずdocstringを書く
- 型ヒントを必ずつける
- エラーハンドリングを入れる
- ログはloggingモジュールを使う（print禁止）
- 1関数50行以内
- 日本語コメントを入れる（自分がパッと見で理解できる程度の平易な表現で）

## ログのフォーマット（Pythonコードで必ずこれを使う）
```python
import logging
logging.basicConfig(
    level=logging.INFO,
    format='%(asctime)s [%(levelname)s] %(message)s',
    handlers=[
        logging.FileHandler('app.log', encoding='utf-8'),
        logging.StreamHandler()
    ]
)
logger = logging.getLogger(__name__)
```

## ファイル・モジュール構成ルール
- 1ファイル = 1つの責務。複数の関心が混在してきたら分割を提案する
- 1ファイルの目安は200行以内（1関数50行ルールと合わせて守る）
- 新しいファイルを作る場合は、作成前にファイル名・配置場所・役割を提示してから実装する
- 推奨構成：
  - `services/`   ビジネスロジック・外部API連携
  - `utils/`      汎用ヘルパー関数（副作用なし）
  - `models/`     データクラス・型定義
  - `handlers/`   イベント・リクエスト処理のエントリポイント

## コードを変更するときのルール
- 既存コードを変更する場合は、変更箇所と理由を先にリストアップしてから実装する
- 変更はタスクに必要な最小限にとどめる。不要なリファクタリングや改善は勝手に行わない
- 既存ファイルの命名規則・スタイルを必ず踏襲する
- 複数ファイルにまたがる変更は、1ファイルずつ順番に行い、都度確認を取る

## デバッグのルール
- 「直して」と頼まれても、まず原因として考えられる箇所を列挙してから修正案を提示する
- 修正後に影響を受ける可能性がある箇所を明示する
- 「直った」と判断する根拠（期待動作の確認方法）を必ず示す
- エラーメッセージが渡された場合は、日本語で原因と対処法を説明してから修正する

## テストのルール（Python）
- 新しい関数を実装したら、必ずそのままpytestのテストも書いて実行する
- テストの最低構成：正常系2・異常系2・境界値1
- テスト実行結果（PASSED/FAILED数）を必ず報告する
- FAILEDがある場合は修正してから次の実装に進む
- テストファイルは tests/ に配置し、test_〇〇.py で命名する

## 品質レビューのルール
- 実装完了後、以下の観点で自己レビューを行い結果を報告する
  - 【重大】セキュリティリスク・データ破壊の可能性・例外の握りつぶし
  - 【中】型ヒント漏れ・docstring漏れ・エラーハンドリング不足
- 重大な問題がなければ「品質チェック: 承認」と明示する
- 既存コードを変更した場合は差分の影響範囲も報告する
